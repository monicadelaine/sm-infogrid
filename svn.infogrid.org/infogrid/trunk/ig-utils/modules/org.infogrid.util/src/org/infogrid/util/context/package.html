Implements a light-weight mechanism for objects to find basic application services.

Many application objects (but not MeshObjects}) in InfoGrid implement the {@link org.infogrid.util.context.ObjectInContext ObjectInContext} interface. An ObjectInContext runs in a {@link org.infogrid.util.context.Context}, which collects a number of environment objects which the ObjectInContext may need. In this respect, it is not all that different from what traditional programming does with environment variables. However, one can only have one set of environment variables per application, but there can be an arbitrary number of Contexts per application, each of which may contain a different set of application services.

A class implementing ObjectInContext is typically instantiated with a Context parameter in the constructor. An ObjectInContext's Context typically does not change during the lifetime of the ObjectInContext. The objects in the Context may change, but not the Context itself.

A Context collects any number of objects, which generally are being found through lookups based on their class or interface.

Contexts can be hierarchical, and InfoGrid makes use of that. If a (child) Context does not hold a requested object locally, it delegates the query to its (parent) Context.

Through configuration of which objects are added to which contexts, one can easily configure large-scale changes to major InfoGrid subsystems with minimum effort. Also, objects can gracefully offer only part of their functionality if certain objects are not present in the environment.

The SimpleContext class provides a simple implementation of the Context interface that is sufficient for most uses.
